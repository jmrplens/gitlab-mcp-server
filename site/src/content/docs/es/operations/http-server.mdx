Por defecto, GitLab MCP Server se ejecuta en **modo stdio** â€” cada cliente de IA inicia su propio proceso de servidor. El **modo HTTP** es una alternativa donde un Ăşnico proceso de servidor atiende a mĂşltiples clientes a travĂ©s de la red, cada uno autenticĂˇndose con su propio token de GitLab.

## CuĂˇndo usar el modo HTTP

| Escenario                                     | Modo Recomendado |
| --------------------------------------------- | ---------------- |
| Desarrollador individual, cliente de IA local | stdio            |
| Equipo compartiendo una instancia de servidor | **HTTP**         |
| Despliegue en servidor remoto/sin pantalla    | **HTTP**         |
| IntegraciĂłn CI/CD con MCP                     | **HTTP**         |
| Pruebas con curl o clientes HTTP              | **HTTP**         |

## Iniciar el servidor

```bash
# Instancia Ăşnica de GitLab (URL por defecto para todos los clientes)
gitlab-mcp-server --http --gitlab-url=https://tu-gitlab.ejemplo.com

# Multi-instancia (cada cliente especifica su URL de GitLab mediante la cabecera GITLAB-URL)
gitlab-mcp-server --http --http-addr=:8080
```

El servidor comienza a escuchar en el puerto 8080 por defecto. El endpoint MCP estĂˇ disponible en `/mcp`.

### Precedencia de configuraciĂłn

Los clientes HTTP solo controlan su token de GitLab y, en modo multi-instancia, el selector `GITLAB-URL`. Las opciones de polĂ­tica del servidor como `--meta-param-schema`, `--rate-limit-rps`, `--read-only`, `--safe-mode`, `--auth-mode` y `--trusted-proxy-header` quedan fijadas por el proceso MCP y no pueden cambiarse por usuario, sesiĂłn ni peticiĂłn JSON-RPC.

Si un cliente envĂ­a cabeceras con aspecto de configuraciĂłn, como `META-PARAM-SCHEMA`, `RATE-LIMIT-RPS` o `GITLAB-SAFE-MODE`, el servidor las ignora y registra sus nombres en `ignored_options` sin registrar sus valores.

## AutenticaciĂłn

Los clientes deben proporcionar su Token de Acceso Personal de GitLab en cada solicitud HTTP usando una de dos cabeceras.

Cuando el servidor arranca sin `--gitlab-url`, los clientes deben especificar a quĂ© instancia de GitLab dirigirse mediante la cabecera `GITLAB-URL`:

```http
GITLAB-URL: https://gitlab.ejemplo.com
```

Si `--gitlab-url` se estableciĂł al iniciar, esta cabecera se ignora y se registra. Si `--gitlab-url` no se estableciĂł y la cabecera se omite, la solicitud se rechaza.

### Cabecera private-token (recomendada)

```http
PRIVATE-TOKEN: glpat-xxxxxxxxxxxxxxxxxxxx
```

### Cabecera authorization Bearer

```http
Authorization: Bearer glpat-xxxxxxxxxxxxxxxxxxxx
```

Si ambas cabeceras estĂˇn presentes, `PRIVATE-TOKEN` tiene precedencia. Las solicitudes sin un token vĂˇlido son rechazadas.

### Modo OAuth

El modo OAuth (`--auth-mode=oauth`) habilita autenticaciĂłn OAuth 2.1 compatible con RFC 9728. En lugar de gestionar tokens manualmente, los clientes MCP descubren el servidor de autorizaciĂłn automĂˇticamente y manejan el flujo OAuth:

```bash
gitlab-mcp-server --http --gitlab-url=https://tu-gitlab.ejemplo.com --auth-mode=oauth
```

**CĂłmo funciona:**

1. El servidor expone `/.well-known/oauth-protected-resource` con metadatos que apuntan a tu instancia de GitLab como servidor de autorizaciĂłn
2. Los clientes MCP (VS Code, Claude Code) descubren este endpoint e inician el flujo OAuth 2.1 PKCE
3. Los usuarios autorizan en el navegador â€” no se requiere copiar tokens
4. El servidor valida los tokens Bearer contra la API de GitLab y cachea la identidad durante `--oauth-cache-ttl` (por defecto: 15 minutos)

## GestiĂłn de sesiones

### Arquitectura del pool de servidores

El nĂşcleo del modo HTTP es un **pool LRU limitado** de instancias de servidor MCP, indexado por el hash SHA-256 del token **y** la URL de GitLab de cada cliente.

```mermaid
graph TD
    subgraph "Arquitectura del Modo HTTP"
        REQ1["Cliente A<br/>Token: glpat-aaa<br/>URL: gitlab.com"] --> HANDLER[StreamableHTTPHandler]
        REQ2["Cliente B<br/>Token: glpat-bbb<br/>URL: gitlab.com"] --> HANDLER
        REQ3["Cliente C<br/>Token: glpat-aaa<br/>URL: self-hosted.ejemplo.com"] --> HANDLER

        HANDLER --> POOL[Pool de Servidores]

        POOL --> ENTRY1["hash(glpat-aaa + gitlab.com)<br/>Servidor MCP + Cliente GitLab"]
        POOL --> ENTRY2["hash(glpat-bbb + gitlab.com)<br/>Servidor MCP + Cliente GitLab"]
        POOL --> ENTRY3["hash(glpat-aaa + self-hosted)<br/>Servidor MCP + Cliente GitLab"]
    end

    ENTRY1 --> GL1["API de GitLab<br/>como usuario A @ gitlab.com"]
    ENTRY2 --> GL2["API de GitLab<br/>como usuario B @ gitlab.com"]
    ENTRY3 --> GL3["API de GitLab<br/>como usuario A @ self-hosted"]
```

**Propiedades clave:**

- Los clientes con el **mismo token y la misma URL de GitLab** comparten la misma instancia del servidor MCP
- Los clientes con **diferentes tokens** o **diferentes URLs de GitLab** obtienen instancias completamente aisladas
- Los tokens sin procesar **nunca se almacenan** â€” solo se mantienen hashes SHA-256 de token+URL en memoria
- Cuando el pool alcanza `--max-http-clients`, la entrada menos usada recientemente es desalojada

### Ciclo de vida de la sesiĂłn

1. **Primera solicitud**: El token y la URL de GitLab se extraen, se combinan y hashean, y se crea un nuevo servidor MCP + cliente GitLab
2. **Solicitudes posteriores**: La entrada existente se encuentra y se promueve en la lista LRU
3. **Timeout de inactividad**: DespuĂ©s de `--session-timeout` de inactividad, la sesiĂłn MCP se cierra (pero la entrada del pool permanece)
4. **Desalojo del pool**: Cuando se alcanza la capacidad, la entrada mĂˇs antigua se elimina completamente

## Rate limiting

El modo HTTP incluye un **rate limiter por servidor con token bucket** opcional que regula las solicitudes `tools/call`. El limitador estĂˇ **desactivado por defecto** (`--rate-limit-rps=0`) y se aplica a cada entrada del pool de forma independiente â€” es decir, el Ăˇmbito es la misma clave `(token + URL de GitLab)` que usa el pool de servidores.

### ConfiguraciĂłn

| Flag                 | Por defecto | Significado                                                                       |
| -------------------- | ----------- | --------------------------------------------------------------------------------- |
| `--rate-limit-rps`   | `0`         | Tasa sostenida de relleno, en solicitudes por segundo. `0` desactiva el limitador |
| `--rate-limit-burst` | `40`        | Capacidad mĂˇxima del bucket (pico de rĂˇfaga durante 1s)                           |

Cuando `--rate-limit-rps > 0`, cada entrada del pool obtiene su propio token bucket dimensionado en `--rate-limit-burst` tokens, rellenado a `--rate-limit-rps` por segundo. Solo las solicitudes `tools/call` consumen tokens; `tools/list`, `resources/*`, `prompts/*`, `initialize` y otras RPCs de bajo coste no estĂˇn limitadas.

### Comportamiento al agotarse

Cuando una solicitud agotarĂ­a el bucket, el servidor devuelve un `CallToolResult` con `IsError: true` y un mensaje de texto como `rate limit exceeded for <tool>; retry after a short backoff`. Los clientes deben aplicar backoff (exponencial o detectando ese mensaje) y reintentar. El limitador no devuelve HTTP 429 porque el lĂ­mite se aplica **despuĂ©s** del enrutado JSON-RPC, dentro de la capa MCP.

### GuĂ­a de dimensionado

- **Despliegue de un solo usuario** (dev local tĂ­pico): dĂ©jalo desactivado (`--rate-limit-rps=0`)
- **Instancia compartida tras un proxy** (Fly.io, Kubernetes): empieza con `--rate-limit-rps=10 --rate-limit-burst=40`. Cada par token+URL obtiene su propia cuota, lo que protege frente a un Ăşnico cliente ruidoso sin afectar a los demĂˇs
- **Despliegue multi-tenant grande**: combina con rate limiting a nivel de infraestructura (Cloudflare, Caddy, nginx). El limitador a nivel MCP es una red de seguridad, no un sustituto del control en el edge

:::caution
Como el limitador hace scoping por token+URL, un cliente que rote tokens frecuentemente puede saltarlo. Usa rate limiting a nivel de infraestructura encima para cuotas duras. El flag `--trusted-proxy-header` es necesario para una aplicaciĂłn precisa por IP en la capa de proxy.
:::

## Despliegue en Fly.io

[Fly.io](https://fly.io/) es una plataforma gestionada que ejecuta imĂˇgenes Docker globalmente con TLS integrado, enrutamiento anycast y escalado de mĂˇquinas por regiĂłn. El repositorio incluye un [`fly.toml`](https://github.com/jmrplens/gitlab-mcp-server/blob/main/fly.toml) de referencia configurado en **modo multi-instancia** â€” cada cliente suministra su propio token de GitLab y la cabecera `GITLAB-URL` por peticiĂłn, por lo que una Ăşnica app de Fly puede servir a usuarios conectados a diferentes instancias de GitLab.

### Requisitos previos

- Una [cuenta de Fly.io](https://fly.io/app/sign-up) y la CLI [`flyctl`](https://fly.io/docs/hands-on/install-flyctl/) instalada
- Un clon del repositorio (o un fork con tu propio `fly.toml`)

### Despliegue inicial

```bash

# 2. Crear la app (usa un nombre Ăşnico; el por defecto en fly.toml es gitlab-mcp-server)

# 3. Desplegar

```

El `fly.toml` incluido usa el `Dockerfile` multi-stage de la raĂ­z del repositorio y sobrescribe el `CMD` del contenedor con flags de modo HTTP:

```toml

    "--http-addr", "0.0.0.0:8080",
    "--meta-tools",
    "--auto-update=false",
    "--trusted-proxy-header", "Fly-Client-IP"

```

:::tip
`--trusted-proxy-header=Fly-Client-IP` es crĂ­tico en Fly.io: el proxy edge de la plataforma establece esta cabecera con la IP real del cliente, y el rate limiter integrado la lee en lugar de la IP del balanceador. No habilites esta cabecera en despliegues accesibles directamente sin un proxy de confianza.
:::

### Comandos operativos


### Salud y TLS

- El proxy de Fly sondea `GET /health` cada 30s con timeout de 5s (configurado en `[[http_service.checks]]`)
- TLS se termina en el edge de Fly con `force_https = true` â€” el trĂˇfico interno a la mĂˇquina en el puerto 8080 es HTTP
- `auto_stop_machines = "stop"` y `min_machines_running = 0` permiten que los despliegues inactivos escalen a cero entre peticiones

### Auto-actualizaciĂłn en Fly

La auto-actualizaciĂłn estĂˇ **deshabilitada** en la configuraciĂłn incluida (`--auto-update=false`). En Fly.io, el flujo de actualizaciĂłn recomendado es redesplegar con una imagen nueva:

Esto reemplaza las mĂˇquinas en ejecuciĂłn con la nueva versiĂłn atĂłmicamente y preserva tus secretos y configuraciĂłn.

### OAuth en Fly.io

El `fly.toml` incluido se ejecuta en **modo de autenticaciĂłn legacy** (PAT por peticiĂłn). El modo OAuth tambiĂ©n estĂˇ soportado pero requiere una URL pĂşblica estable conocida al inicio para que los endpoints de descubrimiento OAuth (`/.well-known/oauth-protected-resource`) anuncien los metadatos correctos.
